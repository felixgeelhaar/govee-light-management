## Decisions Made

- Music mode ids validated as non-negative (`>= 0`), not positive — Govee devices can be zero-indexed; H70B6 Floating Mist = mode id 0. Fix belongs in client `MusicMode.validateModeId` (`< 0`), not the plugin, because the plugin already handled 0 everywhere — only the client value object rejected it.
- Ran the full cross-repo release chain end-to-end (user opted into "merge and release all"): client PR → tag → npm publish → plugin dep bump → plugin PR → tag → marketplace release. Rationale: the plugin bump genuinely can't precede the client npm publish (`npm ci` is lockfile-pinned), so order is forced.
- Customer-facing #250 comment written non-technically (reporter is a non-technical customer) — explained the "first mode in the list was skipped" without API/validation jargon.
- Adopted Agent OS memory system (scaffolded memory/ + wiki/ + AGENTS.md). Seeded files from real session state, not blank templates.
- Kept CLAUDE.md and AGENTS.md non-overlapping: CLAUDE.md = technical reference, AGENTS.md = working style + constraints. No symlink (both exist independently).

## What Went Wrong

- Plugin PR #269 first merge attempt failed: branch was `BEHIND` (dependabot CI bumps #264/266/267 landed after branch base). Branch protection requires up-to-date branches. Had to rebase onto origin/main, force-push, re-wait CI, then merge. Lesson: branch off latest origin/main, or expect a rebase before merge.
- Chained `sleep` + poll loop in one Bash call was blocked by the harness (anti-sleep-chaining guard). Re-ran the poll loop alone; use `until <check>; do sleep N; done` for waits.
